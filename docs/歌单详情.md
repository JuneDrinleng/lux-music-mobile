# 歌单详情

## 当前实际使用的是哪一套
你说的“从 HomeTab 点击歌单后进入的歌单详情页”，当前实际用的是：

- `src/screens/Home/Vertical/Tabs/PlaylistTab.tsx`

它不是旧的独立页面 `src/screens/SonglistDetail/*`，而是 `PlaylistTab` 里的“独立详情模式”。

## 打开链路

### 1. 从 HomeTab 进入本地歌单详情

入口链路：

1. `src/screens/Home/Vertical/Tabs/HomeTab.tsx`
   点击首页歌单卡片后，调用 `global.app_event.openPlaylistDetail(listId)`。
2. `src/event/appEvent.ts`
   `openPlaylistDetail(...)` 会把参数标准化成详情 payload，再发出 `openPlaylistDetail` 事件。
3. `src/screens/Home/Vertical/Main.tsx`
   监听这个事件，把当前 payload 存起来，并挂载一个覆盖层：
   `PlaylistTab standaloneDetail={...}`
4. `src/screens/Home/Vertical/Tabs/PlaylistTab.tsx`
   用 `standaloneDetail` 进入独立详情模式，真正渲染歌单详情内容。

### 2. 从搜索结果进入线上歌单详情

现在歌单搜索结果也已经走同一套新详情页：

1. `src/screens/Home/Vertical/SearchPage.tsx`
   点击歌单结果时调用 `global.app_event.openPlaylistDetail({...})`
2. `src/screens/Home/Vertical/Main.tsx`
   保留当前搜索页状态，在它上面再挂一层 `PlaylistTab` 独立详情模式
3. `src/screens/Home/Vertical/Tabs/PlaylistTab.tsx`
   根据 payload 判断这是“本地歌单”还是“线上歌单”

也就是说，现在 HomeTab 和搜索结果页，最终都落到同一个新歌单详情实现。
并且当详情页是从搜索结果进入时，返回会直接回到搜索结果页，而不是跳回 HomeTab。

## 关键文件职责

- `src/event/appEvent.ts`
  定义歌单详情 payload，并作为页面打开事件总线。
- `src/screens/Home/Vertical/Main.tsx`
  负责详情页覆盖层的挂载、卸载，以及和搜索页的切换关系。
- `src/screens/Home/Vertical/Tabs/HomeTab.tsx`
  本地歌单详情的常规入口。
- `src/screens/Home/Vertical/SearchPage.tsx`
  线上歌单搜索结果的入口。
- `src/screens/Home/Vertical/Tabs/PlaylistTab.tsx`
  实际的详情 UI、数据加载、播放、导入、删除、拖拽排序都在这里。

## PlaylistTab 里怎么区分两种详情

`PlaylistTab` 现在通过 `standaloneDetail` 区分两种模式：

- 本地歌单：`type: 'local'`
- 线上歌单：`type: 'onlineSonglist'`

页面内部会再拆成两类数据流：

- 本地歌单详情：
  依赖 `selectedListId`
- 线上歌单详情：
  依赖 `selectedOnlineDetail`

两个模式共用同一套详情 UI 外壳，但数据来源和可用操作不一样。

## 数据是怎么加载的

### 1. 本地歌单详情

数据来源：

- `src/core/list.ts`
  - `getListMusics(listId)`

加载逻辑：

- `PlaylistTab.tsx` 中的 `loadLocalDetailSongs(...)`
- 先取本地歌单歌曲
- 再写入 `playlistSnapshotCache`
- 同步更新封面、歌曲数、详情列表

本地歌单详情会随着以下变化自动刷新：

- 我的列表变化
- 列表内歌曲变化
- 重排 / 删除歌曲后重新拉取

### 2. 线上歌单详情

数据来源：

- `src/core/songlist.ts`
  - `getListDetailAll(source, id)`

加载逻辑：

- `PlaylistTab.tsx` 中的 `loadOnlineDetailSongs(...)`
- 根据搜索结果带进来的 `source + id` 拉全量歌单歌曲
- 再写入 `playlistSnapshotCache`
- 用搜索结果 payload 里的标题 / 封面 / 作者信息渲染头部

当前线上歌单详情头部信息主要来自搜索结果 payload：

- `name`
- `img`
- `author`
- `desc`
- `play_count`

其中页面上真正展示得最直接的是：

- 标题
- 封面
- 作者 + 歌曲数量

## 页面结构

新歌单详情页主体都在 `PlaylistTab.tsx` 的 `renderDetailScene()`。

### 1. 顶部返回区

包含：

- 返回按钮

职责：

- 关闭当前详情层
- 如果是独立详情模式，会回到上一层 Home / Search 浮层状态

### 2. 头图卡片区

包含：

- 歌单封面
- 歌单标题
- 次级信息

本地歌单会显示：

- 歌曲数量
- 可重命名 / 删除时的操作按钮

线上歌单会显示：

- 作者
- 当前已加载的歌曲数量

### 3. 歌曲列表区

包含：

- 歌曲封面
- 歌名
- 来源标签
- 歌手
- 时长

### 4. 本地歌单专属操作

只有本地歌单会显示这些：

- 重命名歌单
- 删除歌单
- 导入歌曲
- 删除单曲
- 长按拖拽排序

线上歌单详情目前是只读列表，不显示这些本地编辑操作。

## 播放逻辑

### 1. 本地歌单

单曲播放走：

- `handlePlaySong(...)`
- `playListAsQueue(listId, index)`

也就是把本地歌单复制成独立播放队列后，从目标歌曲开始播。

### 2. 线上歌单

单曲播放走：

- `handlePlayOnlineDetailSong(...)`
- `setTempList(queueId, songs)`
- `playList(LIST_IDS.TEMP, index)`

也就是把线上歌单详情当前拿到的歌曲列表塞进临时队列，再从目标歌曲开始播。

## 本地歌单编辑逻辑

这些逻辑都在 `PlaylistTab.tsx`：

- 重命名：
  `updateUserList(...)`
- 删除歌单：
  `removeUserList(...)`
- 导入歌曲：
  `loadImportCandidates(...)` + `addListMusics(...)`
- 删除歌曲：
  `removeListMusics(...)`
- 拖拽排序：
  `updateListMusicPosition(...)`

如果你后面想改“本地歌单详情能编辑哪些东西”，优先看这部分。

## 缓存与过渡

页面内部有一个：

- `playlistSnapshotCache`

当前用途：

- 缓存详情页歌曲快照
- 缓存歌曲数
- 缓存详情封面

它现在同时被用于：

- 本地歌单详情
- 线上歌单详情

这样再次打开同一个详情时，可以先用缓存顶一下，再异步刷新。
